Review Git

  1. Apa itu Version Control System
    2.

    Version control system adalah salah satu kategori dalam tools perangkat lunak yang membantu software team dalam mengelola perubahan source code dari waktu ke waktu.

    Version control system melacak setiap perubahan pada kode yang bilamana terjadi kesalahan, developer dapat kembali pada perubahan sebelumnya untuk membandingkan kode dan memperbaiki kesalahan pada kode tersebut tanpa harus mengganggu team lain yang sedang bekerja dalam satu version control system.

  2. Apa itu Git
    3.

    Git merupakan layanan gratis dan open source version control system yang terdistribusi untuk mengatur segala sesuatu dari project kecil sampai project besar dengan mengandalkan kecepatan dan efisiensi.

  3. Dasar-Dasar Git
    4.

    • Git hanya menyimpan `snapshot` bukan `setiap perubahan` pada project: Git hanya berpikir untuk menyimpan snapshot yang akan ditautkan ke file yang sudah tersimpan.

    • Hampir setiap operasi berada di local: Git tidak perlu terhubung ke server untuk mendapatkan history dan menampilkannya cukup membaca langsung dari database lokal.

    • Git memiliki integritas: Git akan melakukan pengecekan isi dari sebuah file, sehingga gitmengetahui semua file yang sudah dilakukan pengecekan.

    • Git pada umumnya hanya menambahkan data: Setiap perubahan yang di commit pada sebuah file hanya akan menambahkan data ke database git (sulit untuk kehilangan data).

    • Git memiliki tiga status: `committed` (data sudah tersimpan di database), `staged` (menandai pada file yang mengalami perubahan) dan `modified` (perubahan file belum ditandai)

  4. Instalasi Git
    5.

    Berikut ini merupakan cara instalasi Git berdasarkan operating system yang digunakan:

    • Instalasi git pada Linux Fedora bisa menggunakan $ yum install git, sedangkan pada Linux Ubuntu bisa menggunakan $ sudo apt-get install git.

    • Instalasi git pada MacOS bisa menggunakan $ brew install git (pastikan sudah terinstal homebrew).

    • Instalasi git pada Windows bisa dengan mendownload installer git pada laman installer (https://git-scm.com/download/win).

    • Pastikan git sudah terinstal dengan menggunakan perintah $ git --version, maka akan menampilkan pesan versi dari git.

  5. Pengaturan Awal Git
    6.

    Sebaiknya melakukan beberapa konfigurasi sebelum menggunakan git di dalam project, hal ini akan tetap ada walaupun git melakukan update. Berikut konfigurasi awal di dalam git:

    1. Jalankan perintah $ git config --global user.name "Your name", contoh: $ git config --global user.name "Ahmad Yulianto"

    2. Jalankan perintah $ git config --global user.email "Your email", contoh: $ git config --global user.email ahmad_yulianto@mail.com

    3. Jalankan perintah $ git config --list untuk melihat hasil konfigurasinya.

  6. Repositori Git
    7.

    Mendapatkan repository git di dalam project, bisa dengan menggunakan dua cara:

    • $ git init untuk menginisialisasi repository git pada project, kemudian akan membuat folder `.git` yang berisikan semua file repository yang akan dibutuhkan git.

    • $ git clone https://github.com/abdur-rohman2883/learn-git untuk menduplikasi repository yang sudah ada, pada perintah tersebut akan menduplikasi project sesuai dengan namanya dan di dalam project tersebut sudah terdapat folder `.git`.

  7. Siklus Hidup File pada Repositori Git
    8.

    Setiap file pada work directory berada pada dua status: "tracked" atau "untracked". Tracked berarti file yang sudah berada di snapshot terakhir (unmodified, modified dan staged).

    Penjelasan gambar:

    1. Pada saat suatu file baru dimasukkan/dibuat di dalam repositori Git, statusnya adalah untracked.

    2. Ketika terjadi perubahan terhadap file tersebut, semua perubahan masuk ke dalam staging area sehingga status filenya adalah staged. Artinya, Git mengetahui ada perubahan pada file tersebut, tapi perubahan tsb. tidak permanen disimpan pada repositori Git.

    3. Kemudian dari semua perubahan tersebut, kita bisa memilih untuk commit perubahan yang mana saja. Commit di sini mirip seperti fitur save pada Microsoft Word, yaitu merekam snapshot file kita. Lalu status file tersebut berubah dari staged menjadi unmodified.

    4. Jika kita melakukan perubahan terhadap file tersebut, maka status file kita akan berubah dari unmodified menjadi modified.

    5. Jika kita ingin membatalkan perubahan tersebut, kita dapat menggunakan "git reset". Tapi jika kita ingin menyimpan perubahan tersebut ke dalam repositori Git, kita dapat menggunakan "git commit", kembali ke poin ketiga, dst.

  8. Berinteraksi dengan Git
    9.

    Terdapat 2 cara untuk berinteraksi dengan git, yaitu:

    • Menggunakan Command Line Interface (CLI), yaitu dengan menggunakan keyword "git" pada command line, seperti yang telah dicontohkan pada Pengaturan Awal Git dan Repositori Git. Setelah kita menginstall git, kita tidak perlu install software lain untuk menggunakan CLI. Daftar lengkap command line git dapat dilihat pada laman ini: click here

    • Menggunakan Graphical User Interface (GUI), yaitu dengan software lain yang telah disediakan oleh Git maupun oleh pihak ketiga lainnya. Saya lebih memilih menggunakan GitHub Desktop sehingga tidak perlu lagi menghafalkan command git. Daftar lengkap software tersebut dapat dilihat dan diunduh di sini: click here

Review GitHub

  1. Apa itu GitHub
    2.

    GitHub adalah penyimpanan terbesar untuk repositori Git, dan Github adalah pusat kolaborasi untuk pengembangan proyek. Sebagian besar dari semua repositori Git tersimpan di GitHub, dan banyak proyek open source menggunakan Github untuk menyimpan Git, memecahkan masalah, menyusun kode, dan hal-hal lainnya.

    Silahkan kunjungi https://github.com/ untuk melakukan pendaftaran akun GitHub.

  2. Local vs. Remote
    3.

    Dalam penggunaan Git, "local" biasanya mengacu kepada repositori Git yang berada di komputer kita dan tidak terhubung internet. Sedangkan "remote" mengacu kepada repositori Git yang berada di GitHub.

    Kita dapat mendorong/push (seperti upload) local repository Git kita ke server GitHub sehingga orang lain dapat berkontribusi ke dalam proyek kita. Kita juga dapat berkontribusi ke dalam proyek orang lain dengan menarik/pull (seperti download) repository GitHub orang tsb., tentunya harus dengan mekanisme izin terlebih dahulu, yaitu fork atau pull request yang akan dijelaskan pada poin berikutnya.

    Kita dapat melakukan set GitHub repository berupa public atau private. Jika private, maka hanya kita sendiri dan orang-orang yang kita undang yang dapat berkontribusi pada proyek tersebut. Jika public, maka semua orang dapat melihat dan men-download repository kita.

  3. Berkontribusi pada Proyek
    4.

    Berikut ini terdapat 2 cara yang memungkinkan kita untuk berkontribusi pada repository GitHub orang lain.

    1. Forking project: Berkontribusi pada project yang sudah ada yang dimana membutuhkan akses untuk push, cara paling mudah adalah dengan fork dari project tersebut.

    2. Pull Request: Berkontribusi pada project yang sudah ada, namun membutuhkan review dari pemilik atau pengelola project tersebut.

  4. Mengelola proyek
    5.

    Membuat, memelihara dan mengelola project sendiri pada git.

    • Membuat project dimulai dari creating new repository di GitHub sampai push ke GitHub.

    • Adding collaborators pada project agar orang lain bisa bergabung dalam pengembangan project.

    • Managing pull request pada project termasuk review dari pull request dan apply pull request.

    Silakan kunjungi laman ini untuk mempelajari lebih lanjut: https://guides.github.com/activities/hello-world/

Conflict di Repository

Version control system biasa digunakan untuk mengelola kontribusi suatu proyek yang sedang dikerjakan oleh banyak developer. Terkadang beberapa developer mengedit konten yang sama pada saat bersamaan. Jika developer A mengedit suatu code yang sedang diedit oleh developer B, maka conflict dapat terjadi. Untuk mengurangi terjadinya conflict, biasanya developer bekerja pada masing-masing branch yang terisolasi. Kemudian command git merge digunakan untuk menggabungkan branch-branch yang terpisah tersebut dan untuk menyelesaikan conflict yang ada.

  1. Penyebab Conflict
    2.

    Conflict biasanya muncul ketika dua orang telah mengubah baris code yang sama di dalam suatu file, atau ketika seorang developer menghapus file yang sedang dimodifikasi oleh developer lain. Dalam kasus tersebut, Git tidak dapat menentukan secara otomatis mana yang benar. Conflict hanya terjadi pada developer yang sedang melakukan merge, sedangkan developer lain tidak menyadari itu. Git akan menandakan file yang terjadi conflict tsb. dan menunda proses merge-nya. Kemudian developer bertanggung jawab untuk menyelesaikan conflict tersebut.

  2. Menghindari Conflict
    3.

    Conflict dapat dihindari dengan melakukan pembagian tugas lalu membuat branch masing-masing yang berasal dari branch utama, untuk kemudian dilakukan merge. Berikut ini terdapat beberapa tips untuk menghindari merge conflict.

    • Asumsi: pastikan semua anggota tim memahami cara kerja git. Gunakan model branching seperti git-flow.

    • Tips 1: Lakukan perubahan sedikit demi sedikit. Misalkan, buat kesepakatan bahwa pada setiap pull request hanya daapt mengubah maksimal 20 file dengan maksimal penambahan 200 baris code. Jika perubahannya kecil, maka semakin kecil kemungkinan akan terjadi tabrakan. Selain itu, hal ini juga akan memudahkan untuk melihat hasil dari perubahan yang dilakukan.

    • Tips 2: Rebase ke main branch setiap ada perubahan. Ketika main branch mengalami perubahan, lakukan rebase branch yang sedang kita kerjakan dari branch utama. Biasanya, master adalah branch utama, jadi lakukan rebase dengan master setidaknya sehari sekali.

    • Tips 3: Review pull request secepatnya dan merge ke main branch. Buat peraturan misalkan, pull request harus direview dalam 3 hari sejak dibuka.

  3. Penyelesaian Conflict
    4.

    Git memberikan petunjuk untuk menyelesaikan konflik di dalam pesan errornya. Biasanya tertulis "Merge conflict in filename1", jadi kita tau ada masalah pada file tsb. Kemudian tertulis "fix conflicts and then commit the results", sehingga jika kita mengikuti arahan tsb, mengedit filenya, lalu melakukan commit, maka conflict selesai. Sebagai developer, kita menentukan mana code yang harus dipertahankan dan mana code yang harus dihapus/dipindahkan saat terjadi merge conflict.
